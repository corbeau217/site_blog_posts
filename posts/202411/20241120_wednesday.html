  • shapes of programming

    • foundational syntax

      • a game of what is legal to put in a place and what isnt
      • learning the punctuation of the programming language, and what the things do
      • goes from a jigsaw puzzle to a game of sudoku as you learn

    • effects on the surroundings

      • next is to better understand the effects on the surroundings and how things interconnect (conditions, loops, scope, control flow)
      • from the sudoku puzzle it was before, it becomes a game of checkers, planning out your moves in advance

    • modularisation of code

      • learning to modularise your code and recognise sections of repeated data (things that should be functions, classes and objects, or libraries)
      • it becomes more of a focus on the alternative solutions, structure of our solutions, and the efficiency of the code
      • each item now has a purpose and once again they are blocks we use to build a larger picture
      • from checkers we move to a game of chess, planning our moves in advance, keeping in mind the rules of what each piece's limitations are
      • at this point, cost versus reward is what weighs on our actions

    • then comes reliability, reusability, transferability, and separation of concerns

      • now is the focus on composition of our solutions and programs
      • a jigsaw puzzle develops in the sense of data structures, design patterns, and compartmentalisation
      • when a piece of our puzzle becomes to large to get around, we break it into smaller parts so we can focus on each part
      • managing our resources is part of this stage as we learn where to draw the lines in what should and shouldn't be done
      • moving from chess into a turn based strategy
      • it's important to know that one of the resources to manage is burnout from overworking. this takes the reusability/transferability/separation of concerns in many directions following the different philosophies of programming

    • alternative focuses

      • efficiency/reliability found through division of labour

        • (software) - within the code structure by using divide and conquer algorithms
        • (software) - parallelisation [concurrency/multi-threading] of the program
        • (hardware) - the connection between devices to transmit information (digital messaging or peer to peer connections)
        • (hardware) - division of labour between multiple devices (block chains, clouds, hosted information)
      • or instead of efficiency/reliability, we can focus on the requirements and useability of the system (UI/UX design)

  • the main focus of the day was on how vector and matrix operations are performed

    • with the motivation of finding utility in different operations and how they might replace actions which are needlessly inefficient within common actions

    • graphical systems are performing many operations per frame drawn

      • each second - draws a number of frames
      • each frame - draws a number of shapes
      • each shape - draws a number of vertices
      • each vertex - is drawn in groups of 3 as part of a face, processing all information within main function of the vertex shader
      • each face - is drawn as part of the pixels it covers in screen space
      • each pixel - processes all the code within main of the fragment shader
    • due to the volume of things being handles, we rely heavily on parallelisation, but as a result we end up repeating a lot of work
    • minimising the repeated work between each "action" or thing to do, helps to improve the speed (in terms of needing to do fewer operations each frame, meaning our frame takes less time to draw)

    • a method to minimise this is focusing on the operations required per calculation

      • using barycentric interpolation (through the use of varyings) from vertex shader to fragment shader, means that the data is already partially processed for use in the fragment shader
      • using in-built functions like dot products, cross products, matrix multiplication, existing variables/data, and matrix/vector operations, means that our program can use the specialised hardware to perform the calculations in fewer (or faster in clock time) operations
      • understanding what these operations do, will mean that we can just do some cheeky algebraic factorisation and turn it into a game of sudoku/checkers/chess, to make it that much more efficient
      • there was also a very brief thought experiment about what it might mean to redo the entire pipeline using 5D vectors, and 5x5 matrices, with what information would be preserved (is it possible to segregate the rotation and scale? would the 5th value be something relating to eigenvectors/determinants?)
